# Proof-Driven Platform Design Audit Plan

## Objective
Extract reusable UI/UX and design principles from 6 positioning-adjacent sites for autonomous-agent.dev — a platform turning requirements into machine-verifiable, proof-traced delivery.

## Target Sites
1. **Kiro** (https://kiro.dev) - AWS spec-driven IDE
2. **Tessl** (https://tessl.io) - Spec-driven testing
3. **OpenSpec** (https://openspec.dev) - Specification platform
4. **AgentProofs** (https://agentproofs.io) - Proof/attestation platform
5. **Lakera** (https://www.lakera.ai) - AI security/compliance
6. **Mintlify** (https://mintlify.com) - Docs platform (best-in-class docs aesthetic)

## Deliverable Structure (Per Site)

### Profile Components
- **POSITIONING**: Hero headline + subhead (verbatim); brand promise; how they frame specs/proof/compliance/trust
- **IA / NAVIGATION**: Top-level nav; page set; footer structure; marketing ↔ docs relationship
- **COLOR SYSTEM**: Light/dark modes; primary bg; accent/brand hex; palette character
- **TYPOGRAPHY**: Heading + body fonts; monospace usage; notable treatments
- **LAYOUT & DENSITY**: Container width; editorial vs product; section rhythm
- **SIGNATURE MOTIFS**: Spec/requirement visualizations; proof/attestation graphics; compliance badges; code blocks; docs patterns; diagrams
- **CREDIBILITY MODULES**: Compliance badges (SOC2, ISO); customer logos; benchmarks; research; backers; GitHub stats
- **CTA STRATEGY**: Primary + secondary CTA (verbatim); placement patterns
- **STEAL / AVOID**: 2-3 takeaways for requirement-to-evidence traceability platforms selling to regulated enterprises

## Synthesis Output
**CLUSTER SYNTHESIS** (5-8 bullets): How spec/proof/compliance/docs brands convey rigor + trust.
- Dominant patterns across sites
- What proof-first platform should adopt vs avoid
- Specific design recommendations (colors, fonts, section patterns)
- Special focus on docs-site quality (critical for developer platform)

## Notes
- Read-only research only — no file creation beyond plan
- Focus on: specs, proofs, compliance, trust, developer experience
- Look for patterns in: visual hierarchy, trust signals, code/spec block treatment, docs navigation
- Be specific in final report (colors, fonts, section patterns — not vague observations)
