# Autonomous Agent — `autonomous-agent.dev` flagship website build plan

## Context

**Why we're building this.** The repo contains an audit-grade specification for a product — internally
*Spec-to-Evidence Coverage Control System* — but **no public presence**. The owned root domain
`autonomous-agent.dev` has no site. The goal is to build the commercial front door for the flagship
product **Autonomous Agent** (the *Autonomous SDLC Platform*): a single turn-key system that consolidates
the fragmented AI-tooling stack into one closed-loop delivery engine — **"what Wiz did for cloud security,
Autonomous Agent does for the AI software-delivery stack."**

**What this site is — and isn't.**
- It is a **single-flagship, category-defining manifesto site**. The job is to *articulate and own a real,
  industry-wide problem* and position Autonomous Agent as the consolidation that solves it.
- It is **not** a lead-gen funnel (no pricing/demo/sales motion yet) and **not** an umbrella/portfolio site.
  Do **not** reference subdomains or sibling products anywhere in site content.
- Primary actions are soft: *See how it works* / *Read the manifesto* / optional *Get notified*.

**Intended outcome.** A Tier-1-AI-lab-grade site whose centerpiece is an animated, legible visualization of
the closed delivery loop — ending in a fail-closed gate that rejects unproven work — backed by an editorial
manifesto and a proof model no competitor currently claims.

**Grounding.** Direction is grounded in a live Playwright audit of 12 best-in-class exemplars (Neon,
Anthropic, OpenAI, Mistral, xAI, Vercel, Linear, Cognition/Devin, Temporal, Langfuse, Kiro, + gold-standard
patterns) and the product corpus (PRD, requirements, design, market research, formal verification, audit).

## 1. Positioning & the strategic wedge

**Positioning stance (load-bearing — read first).** Autonomous Agent does **not** compete with Tier-1 AI labs
or any point solution in the market. It **stands on top of and elevates** the proven, in-production best-of-breed
solutions — orchestrating and combining them into one turn-key, out-of-the-box system that ships with every best
practice, must-have, and the complete stack required for agent-driven SDLC. It is **not a startup racing rivals;
it is a solution provider for an industry-wide problem**: today, reaching production-grade agentic delivery forces
AI architects to hand-glue 20+ fragmented tools up a steep learning curve, with no guarantee that errors stay out
of production or that every feature/requirement was actually delivered. Autonomous Agent removes that fragmentation
and that human glue-work — and **guarantees complete, evidence-backed delivery.**

> **The thesis (hero + manifesto core line):** *Every one of these problems is already solved — by Claude Code,
> Neon, Temporal, Langfuse, Playwright, Semgrep, and a dozen other proven Tier-1 tools. What no one has done is
> wire them all together, gate them correctly, and ship it as a product. That is what this is.*

in-production components rather than reinventing them:
- **Coding agent / substrate:** Claude Code (subagents, hooks, plan mode, worktrees)

rationale), zero silently unaddressed** — is itself a proof artifact the site can show honestly. Every pain
point should trace to a section/claim and, ideally, to the built-in capability that resolves it.

**Governing invariant (the soul of the brand, verbatim from the spec):** deterministic gates (hooks, CI, OPA)
decide completeness from verifiable facts only — model self-assessment *informs, never gates*.

**The "what it replaces" table** (ready-made homepage content from the brief): human-as-memory →

## 2. Competitive design audit — what to adopt vs. avoid

**Universal Tier-1 grammar (table stakes):** custom/branded typeface; oversized display (48–68px) with tight
negative tracking (−1.4 to −2.7px); monochrome base + exactly one signature accent; enterprise logo-wall /
stat-bar credibility; research-forward nav.

**The "make a complex concept digestible" device — every leader has one:** Neon (generative WebGL data-
spectrum = compute made alive); Temporal (live code-block of the actual durable loop); xAI/Linear/Kiro
(realistic product-UI + agent-execution-trace cards); Mistral (generative color art). **Our version is the
animated closed loop + evidence trace** (Section 4).

**Adopt:** near-black-not-pure-black canvas (Linear `#08090A`); thin-elegant oversized display; one accent;
mega-menu sublabels for multi-capability products (Vercel); credibility stat-bar (Langfuse); manifesto +
numbered sections + blueprint rules (Cognition); machine artifacts as monospace.
**Avoid:** decorative AI-glow gradients with no meaning; generic Inter-only with no point of view; alarm-red
used as decoration; over-busy product screenshots before the concept is established.

## 3. Brand & design system (the confirmed direction)

**Art direction:** *Dark, premium, motion-as-information.* Reads as a serious research lab AND delivers
cinematic depth. **Hard rule: every animated element is a state machine, not a CSS loop** — proof traces
pulse, coverage indicators fill, gate checks close. If a motion isn't communicating a product state, it does
not ship. One accent, reserved for verification states and system signals — never decoration.

### Color tokens (dark, restrained, semantic)
| Token | Value (proposed) | Use |
|---|---|---|
| `--canvas` | `#0A0B0D` | page base (between Linear `#08090A` and pure black) |
| `--surface` | `#121417` | elevated cards/panels |
| `--surface-2` | `#181B1F` | nested panels, code/trace blocks |
| `--border` | `#1E2227` | hairline rules, blueprint grid |
| `--text` | `#F5F7FA` | primary |
| `--text-muted` | `#8A8F98` | secondary (Linear's muted) |
| `--text-faint` | `#5A606B` | tertiary, labels |
| `--proven` | `#34E1A0` *(signature accent)* | **verified / evidence / system-correct** signals ONLY |
| `--pending` | `#E2B340` | awaiting-evidence state (sparing) |
| `--signal` | `#34E1A0` | the single brand accent (== `--proven`) |

> **Accent discipline:** the accent means "proven / system working." Crucially, when the gate **closes on
> unproven work, that is the system succeeding** — show it as the accent doing its job, not as alarm-red.
> Unproven items dim to `--text-faint`/`--pending`; reserve red strictly for genuine error states (rare).

### Typography
- **Display/headings:** clean grotesk, thin-to-medium weight (400–600), tight tracking (−1.5 to −2px), large.
  Recommend **Geist Sans** (open, dev-native, Vercel-tier) or Inter Variable as fallback. Optional upgrade: a
  characterful licensed grotesk (Söhne/ABC Diatype) if budget allows.
- **Body:** Geist Sans / Inter, 17–19px, generous line-height.
- **Mono (load-bearing):** **Geist Mono / JetBrains Mono / Berkeley Mono** — used for *all machine artifacts*:
  requirement IDs, evidence records, hashes, gate verdicts, traces. Mono = "this is verifiable fact."

### Motion system
- **Rive** — the hero closed-loop state machine (deterministic, scroll/interaction-triggered, performant).
- **Framer Motion** — informational micro-interactions only (coverage fills, trace reveals tied to meaning).
- **`prefers-reduced-motion`** — every animation has a meaningful static fallback (the loop renders as a
  legible labeled diagram).

### Layout
- Generous containers (max ~1200–1280px), blueprint hairline rules (Cognition), numbered manifesto sections,
  asymmetric editorial grid, large vertical rhythm.

## 4. The signature centerpiece — the animated closed loop (Rive)

The hero IS the product's runtime in motion: **Intent → Spec → Decompose → Implement → Gate → Verify → Prove**,
cycling to a fail-closed gate. Built in Rive so each node behaves like a machine state, not a keyframe.

**The positioning, rendered literally (critical).** Every node names the **proven Tier-1 tool underneath it** —
revealed *as the loop plays*, not as a logo parade, but as proof the system is *wired from trusted components,
not invented from scratch*. **You see the loop, you see the tools it orchestrates, you see the gate close —
claim proven in five seconds.**

| Loop node | Tier-1 tool revealed underneath |
|---|---|
| **Intent** | the human brief (the only human input) |
| **Spec** | Claude Code spec compiler → EARS + default-unproven `feature_list.json` |
| **Decompose** | Claude Code subagents + git worktrees (one bounded slice at a time) |
| **Implement** | Claude Code (the coding agent) |
| **Gate** | Claude Code hooks (per-edit) · OPA/Conftest · GitHub rulesets (fail-closed) |
| **Verify** | Playwright · Semgrep/CodeQL/SonarQube · Hypothesis · k6/Lighthouse · axe-core + independent verifier |
| **Prove** | SLSA provenance · hash-chained audit log · Z3 (logic model) — surfaced via Langfuse/OTel |
| *substrate (under the whole loop)* | **Neon** (durable state) · **Langfuse/OTel** (live trace) · **Temporal** (optional durable outer loop) |

> **Build note:** the gate operates at **two moments** — mid-flight per-edit enforcement (Claude Code hooks) and
> the climactic **completion gate** (fail-closed: Stop hook + OPA + GitHub ruleset). The "gate closing" on
> unproven work is the emotional climax — it must feel *satisfying, the system doing its job*, never an error.

It must accomplish three things simultaneously (per the confirmed brief):

1. **Legible in < 5 seconds** — stages read in a clear left-to-right cycle without study.
2. **The fail-closed gate is visceral** — when a requirement reaches the gate *without evidence*, something
   stops/closes/rejects. The rejection feels **satisfying, not alarming — it is the feature.** (Gate glows the
   accent as it correctly holds the line; the unproven item is held back, not error-flashed.)
3. **Requirement IDs & evidence traces are live machine artifacts** — monospaced `REQ-014`, evidence records
   (`test_file · test_name · output_hash · collected_at`) that **light up to `--proven` as they get proved** —
   never hand-drawn icons or marketing copy.

**Secondary live visual:** an **evidence/audit-trace panel** — a monospaced, tamper-evident execution trace
(requirement-ID-tagged spans, hash-chained entries) that demonstrates "an auditor could verify this," echoing
observability trace UIs (Langfuse) but framed as *proof*, not just telemetry.

**First execution deliverable** is a clickable/animated prototype of this centerpiece (the brand lives or dies
on its craft) — validated before the rest of the site is built out.

## B.1 Requirements index — all 189 (EARS one-liners)

### DS — Brand & Design System
- **DS-01** `[P0]` The design system SHALL define a complete dark-theme color token set (--canvas #0A0B0D, --surface #121417, --surface-2 #181B1F, --border #1E2227, --text #F5F7FA, --text-muted #8A8F98, --text-faint #5A606B, --proven #34E1A0, --pending #E2B340, --signal aliased to --proven) as the single source of truth, and every rendered surface, text, and border on the site SHALL resolve its color from a named token rather than a hard-coded literal.
- **DS-02** `[P0]` WHERE a token pairing is used for text or meaningful non-text UI, the design system SHALL guarantee WCAG 2.2 AA contrast (>=4.5:1 normal text, >=3:1 large text and meaningful graphical objects/focus indicators) against its intended background token, and IF a proposed token fails its target ratio, THEN the system SHALL adopt an AA-corrected value before launch.
- **DS-03** `[P0]` The design system SHALL define a typographic system using a self-hosted display/body grotesk (Geist Sans, Inter Variable fallback) at weights 400-600 with tight negative tracking (-1.5px to -2px on display sizes), plus a modular type scale, fluid responsive sizing, and consistent line-heights for display, heading, body (17-19px), label, and caption roles.
- **DS-04** `[P0]` The design system SHALL self-host all brand fonts (display/body grotesk and the monospace family) as subset, preloaded WOFF2 files served same-origin with no third-party font requests, and WHEN a web font is still loading, the system SHALL apply font-display:swap with a metric-matched fallback so that cumulative layout shift from font swap is negligible.
- **DS-05** `[P0]` WHERE content is a machine artifact (requirement IDs e.g. REQ-014, evidence records with test_file / test_name / output_hash / collected_at, content hashes, gate verdicts, code, and execution-trace spans), the design system SHALL render it in the monospace family (Geist Mono / JetBrains Mono) with tabular numerals and SHALL never render such artifacts in the display/body grotesk.
- **DS-06** `[P0]` The design system SHALL reserve --proven/--signal as the single accent used ONLY for verification states and system-correct signals (proved requirements, satisfied gates, evidence confirmation, focus/active system indicators), and IF an element is purely decorative or marketing chrome, THEN the system SHALL NOT use the accent for it.
- **DS-07** `[P0]` The design system SHALL define spacing (base-grid scale), border-radius, elevation/shadow, hairline-border, and container-width tokens (max content width ~1200-1280px), and every component SHALL compose layout exclusively from these tokens.
- **DS-08** `[P0]` The design system SHALL define motion tokens (duration scale, easing curves, and named state-transition semantics) and SHALL require that every shipped animation map to a real product/UI state transition; IF a proposed motion does not communicate a verifiable state (proof pulsing, coverage filling, gate closing, evidence lighting up), THEN the system SHALL NOT ship it.
- **DS-09** `[P0]` WHEN the user agent reports prefers-reduced-motion: reduce, the design system SHALL disable non-essential motion and present a legible static equivalent that conveys the same state information (e.g. the closed loop renders as a labeled static diagram; coverage indicators show final filled state; gate shows its held verdict).
- **DS-10** `[P0]` The design system SHALL maintain a single token source of truth (DESIGN.md as the human contract plus Style Dictionary JSON as the machine source) that builds deterministically into CSS custom properties and the Tailwind v4 theme, and the system SHALL fail the build IF the generated CSS/Tailwind theme drifts from the token source.
- **DS-11** `[P0]` The design system SHALL provide a documented library of token-driven component primitives (button variants for soft actions, cards/panels, status badges, and navigation) with defined default, hover, active, focus-visible, and disabled states, and these primitives SHALL be the only sanctioned building blocks for site UI.
- **DS-12** `[P0]` The design system SHALL define a code/trace block primitive rendered on --surface-2 with the monospace family, line/column alignment, optional requirement-ID gutter and hash-chain affordance, and accessible copy/scroll behavior, and WHEN an evidence record or gate verdict is displayed it SHALL surface its fields (test_file, test_name, output_hash, collected_at, verdict) as labeled monospaced data.
- **DS-13** `[P0]` The design system SHALL guarantee that all interactive primitives are keyboard-operable with a visible focus indicator, that state is never conveyed by color alone, that semantic roles/landmarks/labels are present, and that the system honors WCAG 2.2 AA including target size and focus-not-obscured.
- **DS-14** `[P1]` The design system SHALL define the 'Autonomous Agent' wordmark and logo usage (sizing, clear-space, color-on-dark, monochrome and single-accent variants, favicon/app-icon, and the 'Autonomous SDLC Platform' category descriptor lockup) and a minimal, consistent line-icon set, and IF an icon would imply an unverifiable claim or decorative AI-glow, THEN the system SHALL exclude it.
- **DS-15** `[P1]` The design system SHALL operate within documented performance budgets such that critical CSS/font/token payloads and motion assets do not regress Core Web Vitals (LCP < 2.5s, CLS < 0.1, INP < 200ms) on a mid-tier device, and WHILE the hero animation loads it SHALL not block the main thread or cause layout shift.
- **DS-16** `[P0]` The design system SHALL NOT provide visual treatments (badges, counters, hero stats, mono 'verdict' chips) that present a specific quantified verification claim (e.g. an 'N/N formally verified' figure) until that figure is corrected and independently verifiable, and WHERE a verification state is shown it SHALL render only states that are evidence-backed, never a self-graded summary number.
- **DS-17** `[P1]` WHILE a token-driven component or motion asset is loading, the design system SHALL present a calm skeleton/placeholder using --surface/--surface-2 tokens, and IF an enhancement asset (Rive runtime, font, trace data) fails to load, THEN the system SHALL fall back to the static token-styled equivalent without alarm-red decoration.
- **DS-18** `[P1]` The design system SHALL define responsive breakpoints and behaviors such that typography scales fluidly, containers and spacing adapt, navigation collapses accessibly, and the hero/trace degrade to a clean labeled diagram on small screens, with no horizontal overflow from 320px up.

### IA — Information Architecture & Navigation
- **IA-01** `[P0]` The site SHALL expose exactly the following canonical routes at the root domain autonomous-agent.dev: / (home), /how-it-works, /proof, /manifesto, /writing, /docs. WHERE /writing or /docs is not launched with content, the site SHALL still resolve the route to a designed stub page rather than a hard 404.
- **IA-02** `[P0]` The site SHALL serve every page at a single canonical, lowercase, hyphen-delimited, extensionless URL on the https://autonomous-agent.dev origin. WHEN a request arrives with a non-canonical casing, trailing slash, or duplicate slash, the site SHALL issue a 308 permanent redirect to the canonical URL.
- **IA-03** `[P0]` The primary navigation SHALL present only the wordmark and the destinations How it works, Proof, Manifesto, and Writing, plus a single soft, low-emphasis Get notified action. The navigation SHALL NOT contain any pricing, demo, contact-sales, sign-in, or purchase affordance.
- **IA-04** `[P0]` WHILE the viewport is at or above the desktop breakpoint, the navigation SHALL display all destinations inline in a single fixed/sticky bar. WHILE the viewport is below the desktop breakpoint, the navigation SHALL collapse non-wordmark destinations into an accessible disclosure (hamburger) menu that opens a focus-trapped panel.
- **IA-05** `[P1]` WHEN a navigation destination corresponds to the current route, the navigation SHALL mark that item as current via aria-current="page" and a restrained visual treatment. WHILE the user scrolls, the navigation SHALL change state (e.g. condense / gain a hairline border or scrim) only to preserve legibility, and any such transition SHALL be an informational state change, not a decorative loop.
- **IA-06** `[P1]` The home page and long pages SHALL assign a stable, semantic anchor id to each major narrative section (e.g. #the-problem, #closed-loop, #proof-model). WHEN a user activates an in-page anchor link, the page SHALL scroll to the target section, update the URL fragment, and move keyboard focus to the section heading. WHERE prefers-reduced-motion is set, the page SHALL jump to the anchor without smooth-scroll animation.
- **IA-07** `[P1]` WHERE a page is below the top level of the information architecture (e.g. a /writing/<slug> article), the page SHALL render a breadcrumb trail from Home to the current page. The breadcrumb SHALL mark the current page as non-interactive with aria-current="page" and SHALL NOT render on the home page.
- **IA-08** `[P0]` IF a request resolves to no known route or content, THEN the site SHALL return an HTTP 404 status with a branded not-found page that preserves the global navigation and offers links back to the canonical primary destinations.
- **IA-09** `[P1]` IF an unrecoverable server or rendering error occurs, THEN the site SHALL return an HTTP 500 status with a branded error page that contains no stack trace, environment detail, or internal identifier, and SHALL offer a path back to the home route.
- **IA-10** `[P0]` The site SHALL enforce a single canonical origin: WHEN a request arrives at the www host, an http scheme, or a registered legacy/alias path, the site SHALL 301/308 permanently redirect to the corresponding https://autonomous-agent.dev canonical URL. The redirect map SHALL be a declared, version-controlled table.
- **IA-11** `[P1]` The site SHALL generate a sitemap.xml from the route manifest containing only canonical, indexable URLs, and SHALL serve a robots.txt that references the sitemap. WHERE a route is a non-content stub (/docs, empty /writing) or an error/utility page, the site SHALL exclude it from the sitemap and mark it noindex.
- **IA-12** `[P0]` The global footer SHALL provide secondary navigation to all canonical destinations, a single soft Get notified email capture, and a restrained trust/security note. The footer SHALL NOT display any specific 'N/N formally verified' count or other quantitative claim that is not independently evidence-backed.
- **IA-13** `[P0]` The site SHALL expose semantic landmark regions (banner/header, main, navigation, contentinfo/footer) on every page, SHALL provide a visible-on-focus skip-to-content link as the first focusable element, and WHEN a route change completes, the site SHALL move focus to the main content region or page heading.
- **IA-14** `[P1]` WHEN a user navigates between routes, the site SHALL render the destination via SSG/ISR with no full-page reload flash, and WHILE a route or async section is loading, the site SHALL present a defined loading state (skeleton/route-level loading UI) rather than a blank screen or layout jump.
- **IA-15** `[P2]` WHERE /docs is a launch-time placeholder or /writing has no published entries, the site SHALL render a designed stub/empty state that states the section's intent, preserves global navigation, and offers the soft Get notified action — without fabricating content or implying availability.
- **IA-16** `[P1]` Each canonical route SHALL define unique, accurate metadata — title, description, canonical URL, and Open Graph/Twitter card — derived from the route manifest and page content, and SHALL NOT include any unverifiable quantitative claim in share metadata.

### LOOP — Rive Closed-Loop Centerpiece + Evidence Trace
- **LOOP-01** `[P0]` The ClosedLoop centerpiece SHALL render a single continuous closed loop of six named, ordered stages — intent, decompose, implement, verify, prove, gate — laid out so that the traversal direction is unambiguous and the loop visibly returns to its origin. WHEN the centerpiece mounts and motion is permitted, the ClosedLoop SHALL drive the active-stage indicator around the loop in canonical order at a pace that completes one full traversal in 8-14 seconds.
- **LOOP-02** `[P0]` The ClosedLoop SHALL be driven by a single Rive state machine whose named states correspond to the loop stages and whose named inputs and triggers are the only mechanism by which the host application changes what is displayed. The ClosedLoop SHALL NOT use time-based CSS animations or Framer Motion to animate the loop's stage progression, gate behavior, or evidence-proving transitions.
- **LOOP-03** `[P0]` The ClosedLoop SHALL satisfy three concurrent acceptance goals on every default playthrough: (1) it SHALL be legible in under 5 seconds; (2) it SHALL stage the fail-closed gate rejection to feel satisfying, not alarming; and (3) it SHALL render requirement IDs and evidence traces as live monospaced machine artifacts that light up to the proven accent as they are proved. WHERE any one of these three goals cannot be met for a given viewport or motion preference, the ClosedLoop SHALL degrade to the static fallback rather than ship a partial animation that violates a goal.
- **LOOP-04** `[P0]` WHILE the loop runs, the ClosedLoop SHALL display one or more requirement tokens (monospaced, e.g. REQ-014) that visibly traverse the stages, and WHEN a token reaches the prove stage with a complete evidence record, the ClosedLoop SHALL transition that token's state from pending to proven, lighting it to the proven accent. The ClosedLoop SHALL render requirement IDs only in the monospace typeface and SHALL NOT substitute hand-drawn icons or marketing labels for machine artifacts.
- **LOOP-05** `[P0]` The EvidenceTrace panel SHALL render a live, append-only, monospaced stream of evidence records where each proven requirement carries the four-field evidence schema (test_file, test_name, output_hash, collected_at) and entries are visually presented as a tamper-evident, hash-chained sequence. WHEN the loop proves a requirement token (LOOP-04), the EvidenceTrace SHALL append the corresponding evidence record in lockstep and light its requirement ID and output_hash to the proven accent.
- **LOOP-06** `[P0]` WHEN a requirement reaches the gate stage without a complete evidence record, the gate sub-machine SHALL transition to a holding/closed state in which the gate element glows the proven accent to signal a correct system action, and the unproven item SHALL be visibly held back (dimmed toward text-faint/pending) rather than error-flashed. The gate SHALL NOT use alarm-red, shake, buzzer-like motion, or other failure-coded affordances as decoration for a correct rejection.
- **LOOP-07** `[P0]` The gate sub-machine SHALL stage the rejection as an ordered, multi-beat sequence — approach, evaluate (gate-evaluating), hold/close with accent glow, then settle — with each beat driven by a named state, so the moment reads as intentional choreography. WHILE the gate is in the evaluating state, the ClosedLoop SHALL briefly surface the reason the item is held (e.g., a monospaced 'no evidence' / 'unproven' marker) so the rejection is explained, not merely shown.
- **LOOP-08** `[P0]` IF the user agent reports prefers-reduced-motion: reduce, OR IF the Rive runtime fails to load, THEN the ClosedLoop and EvidenceTrace SHALL render a static, fully-labeled diagram of the closed loop (all six stages, the gate in its holding state, at least one proven and one held requirement token) and a static set of representative evidence records, conveying the same information as the animated version. The static fallback SHALL be the server-rendered default so it is present before and without any client JavaScript.
- **LOOP-09** `[P0]` The ClosedLoop and EvidenceTrace SHALL use the proven accent (--proven #34E1A0) exclusively for verified/evidence/system-correct signals — proved tokens, the gate correctly holding, and confirmed evidence hashes — and SHALL use --pending only for awaiting-evidence states and --text-faint for held/unproven items. The centerpiece SHALL NOT introduce any additional saturated hue and SHALL reserve alarm-red strictly for genuine runtime error states (not for the gate's correct rejection).
- **LOOP-10** `[P1]` WHERE interaction is enabled and motion is permitted, the ClosedLoop SHALL allow the visitor to pause/resume the loop and to focus an individual stage or requirement token via pointer and keyboard, driving the underlying Rive inputs deterministically. WHEN the visitor focuses a stage or token, the EvidenceTrace SHALL highlight the corresponding evidence record (and vice versa), keeping the two surfaces synchronized. The loop SHALL auto-resume cinematic playback after a bounded idle period if the visitor takes no further action.
- **LOOP-11** `[P0]` The ClosedLoop and EvidenceTrace SHALL adapt responsively: on large viewports they SHALL present the loop and the live evidence panel side-by-side; on small viewports the ClosedLoop SHALL reflow to a vertically stackable, simplified-but-labeled loop with the evidence trace below, preserving all six stage labels and the gate's holding semantics. IF the viewport is too small to animate the full loop legibly, THEN the ClosedLoop SHALL fall back to the static labeled diagram (per LOOP-08) rather than render an illegible animation.
- **LOOP-12** `[P0]` The ClosedLoop SHALL expose a text equivalent of the loop (an ordered description of the six stages, the gate's fail-closed behavior, and at least one proven and one held requirement) to assistive technology via appropriate roles, names, and descriptions, and the EvidenceTrace SHALL expose its records as readable, navigable text (not canvas-only). WHILE animation plays, the ClosedLoop SHALL NOT convey any information by motion alone that is not also available as text or static state.
- **LOOP-13** `[P0]` The ClosedLoop and EvidenceTrace SHALL render only artifacts drawn from a committed, illustrative fixture that is clearly modeled on the product's real evidence schema, and SHALL NOT display any specific aggregate 'N/N verified' or 'N formally verified' count that is not independently verifiable. WHERE a numeric proof statistic would otherwise appear, the centerpiece SHALL instead show the per-requirement evidence mechanism (the four-field record and the gate verdict) rather than an unbacked headline number.
- **LOOP-14** `[P0]` The ClosedLoop SHALL lazy-load the Rive runtime and the .riv asset without blocking first paint, SHALL reserve its layout box ahead of load to avoid layout shift, and SHALL keep animation work off the critical path so that Core Web Vitals targets are met. IF the device is low-powered, on a save-data/slow connection, or the asset exceeds its load-time budget, THEN the ClosedLoop SHALL present the static fallback instead of the animated canvas.
- **LOOP-15** `[P1]` The ClosedLoop SHALL begin playback only when its container intersects the viewport and SHALL pause playback WHILE the centerpiece is scrolled out of view or the document is hidden (tab backgrounded), resuming from a coherent state when it returns to view. The ClosedLoop SHALL NOT autoplay before the visitor has scrolled it into view on initial load.
- **LOOP-16** `[P1]` WHILE the Rive asset or evidence fixture is loading, the ClosedLoop SHALL display a calm, on-brand skeleton that already shows the loop's labeled structure in a neutral (not-yet-proven) state, and the EvidenceTrace SHALL show a labeled empty/awaiting state rather than a blank panel. IF loading exceeds the budget or fails, THEN the centerpiece SHALL resolve to the static fallback (LOOP-08) and the evidence panel SHALL show a representative static record set, never an indefinite spinner.

### TECH — Tech Stack & Front-end Architecture
- **TECH-01** `[P0]` The front-end application SHALL be built on Next.js App Router with all routes under a single `app/(site)/` route group, each public route (`/`, `/how-it-works`, `/proof`, `/manifesto`, `/writing`, `/docs`) implemented as a `page.tsx` segment, and shared chrome provided by a single root `layout.tsx`. WHERE a route segment can fail, render slowly, or return no content, the application SHALL colocate `error.tsx`, `loading.tsx`, and (where applicable) `not-found.tsx` with that segment.
- **TECH-02** `[P0]` The application SHALL compile under TypeScript `strict` mode with `noUncheckedIndexedAccess` and `exactOptionalPropertyTypes` enabled, and SHALL define explicit types for the evidence-record shape (`{ test_file: string; test_name: string; output_hash: string; collected_at: string }`), requirement-ID strings, loop-state-machine state names, and MDX frontmatter. IF any type error exists, THEN the production build SHALL fail.
- **TECH-03** `[P0]` The application SHALL use Tailwind CSS v4 with its theme configured exclusively from CSS custom properties emitted by Style Dictionary from `design/tokens/*.json`, and SHALL expose semantic tokens (`--canvas`, `--surface`, `--surface-2`, `--border`, `--text`, `--text-muted`, `--text-faint`, `--proven`, `--pending`) plus typographic tokens for the Sans and Mono families. WHERE a component renders a machine artifact (requirement ID, evidence record, hash, gate verdict), it SHALL apply the mono token; WHERE a component needs the brand accent, it SHALL reference `--proven`/`--signal` only.
- **TECH-04** `[P1]` The application SHALL render `/manifesto` and `/writing/[slug]` content from MDX files in `content/`, compiled at build time with a fixed allow-list of MDX components (numbered sections, blueprint rules, pull-quotes, `<EvidenceRecord>`, `<RequirementId>`) and typed, schema-validated frontmatter. IF an MDX file references a component outside the allow-list or fails frontmatter validation, THEN the build SHALL fail.
- **TECH-05** `[P0]` The application SHALL integrate the Rive centerpiece via `@rive-app/react-canvas` inside a single client-boundary wrapper component (`components/loop/ClosedLoop.tsx`) that is dynamically imported with SSR disabled, lazy-loaded when the hero approaches the viewport, and rendered in a reserved fixed-aspect container. WHILE the `.riv` asset is loading, the wrapper SHALL show the static labeled-diagram fallback in the reserved space so that no cumulative layout shift occurs.
- **TECH-06** `[P0]` The application SHALL restrict Rive to the hero closed-loop state machine and any secondary product-state visual, and SHALL restrict Framer Motion to informational micro-interactions whose animated state maps to a real product state (coverage fills, evidence-trace reveals, gate verdicts). WHERE a motion does not communicate a product state, the application SHALL NOT animate it.
- **TECH-07** `[P0]` WHEN the user agent reports `prefers-reduced-motion: reduce`, the application SHALL render the closed loop as a static, fully-labeled diagram conveying the same intent→decompose→implement→verify→prove→gate sequence, SHALL freeze Framer Motion micro-interactions to their final informational state, and SHALL not autoplay the Rive timeline. The static fallback SHALL be legible in under 5 seconds and expose text equivalents for all machine artifacts.
- **TECH-08** `[P0]` The application SHALL statically render (SSG) all marketing routes at build time and SHALL apply Incremental Static Regeneration (ISR) only to MDX-backed content routes (`/manifesto`, `/writing`, `/writing/[slug]`) via a bounded revalidation window. WHERE a route has no per-request dynamic data, the application SHALL NOT opt into dynamic (per-request) rendering.
- **TECH-09** `[P0]` The application SHALL wrap the Rive centerpiece and any client data-dependent region in React error boundaries that render the on-brand static fallback, and SHALL provide segment-level `error.tsx` for every route group. IF a client component throws or an asset fails to load, THEN the boundary SHALL contain the failure, render a legible fallback, and report the error without crashing the rest of the page.
- **TECH-10** `[P1]` WHERE a route or region depends on content that may be absent or still resolving, the application SHALL provide a route-level `loading.tsx` skeleton and a purpose-written empty state. WHILE content is loading, the application SHALL show a token-styled skeleton that matches final layout dimensions; WHEN a collection (e.g. `/writing`) has zero published items, the application SHALL render an intentional empty state rather than a blank list.
- **TECH-11** `[P0]` The application SHALL implement a responsive layout from a 320 px minimum width through ultra-wide, using a max content container of ~1200–1280 px, and SHALL degrade the Rive centerpiece to a clean labeled diagram (vertical or simplified arrangement) below a defined breakpoint where the full animation would be illegible. WHERE viewport width is below the centerpiece breakpoint, the application SHALL prioritize legibility of stages and machine artifacts over animation fidelity.
- **TECH-12** `[P1]` The application SHALL keep state on the server / in route segments by default and SHALL introduce client state only inside explicitly-marked `'use client'` boundaries for genuine interactivity (scroll progress, Rive state, theme/reduced-motion preference, form input). WHERE shared cross-component client state is required, the application SHALL use lightweight React context or a minimal store rather than a global app-wide state framework.
- **TECH-13** `[P0]` The application SHALL treat every quantitative or verification claim rendered in the UI as data carrying a source reference, and SHALL provide a single claims registry that the build validates. IF a claim is rendered without an associated, resolvable source entry, THEN the build SHALL fail; the application SHALL NOT hard-code a specific 'N/N verified' count anywhere until that count is corrected and marked verifiable in the registry.
- **TECH-14** `[P1]` The application SHALL self-host the display/body family (Geist Sans or Inter Variable) and the monospace family (Geist Mono / JetBrains Mono) via `next/font` (local), exposing them as CSS-variable font tokens with size-adjusted fallbacks. WHILE web fonts are loading, the application SHALL avoid layout shift by using `font-display: swap` with metric-compatible fallbacks, and machine artifacts SHALL always resolve to the mono family.
- **TECH-15** `[P1]` The application SHALL emit security headers (a strict Content-Security-Policy, HSTS, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`, frame-ancestors deny) via middleware/edge configuration, SHALL serve the Rive WASM and `.riv` assets under a CSP that permits `wasm-unsafe-eval` only as scoped as required, and SHALL optimize images and static assets. WHERE the CSP would block a required runtime (Rive WASM, self-hosted fonts), the policy SHALL grant the narrowest sufficient allowance and nothing broader.
- **TECH-16** `[P0]` The application SHALL enforce, as required CI gates, performance budgets (Lighthouse performance/SEO/best-practices ≥ 95, LCP < 2.5 s, CLS ≈ 0, bounded JS bundle), accessibility (axe-core zero critical, WCAG 2.2 AA contrast for `--proven` on `--canvas` and all text tokens, full keyboard operability), and motion integrity (every animation maps to a registered product state; reduced-motion fallbacks render). IF any budget is exceeded, THEN the merge SHALL be blocked.

### TOOL — Tooling, Pipeline & CI/CD
- **TOOL-01** `[P0]` The token pipeline SHALL compile a single canonical token source (design/tokens/*.json governed by DESIGN.md) via Style Dictionary into (a) CSS custom properties on :root, (b) a Tailwind v4 @theme layer, and (c) a typed TypeScript token map. WHEN a token source file changes, the pipeline SHALL regenerate all three outputs deterministically from the same input. IF any generated output is committed in a state that does not match a fresh build of the source, THEN the CI token-drift gate SHALL fail the build.
- **TOOL-02** `[P0]` WHEN tokens are built, the pipeline SHALL compute contrast ratios for every documented foreground/background pairing and SHALL fail the build IF any text pairing falls below WCAG 2.2 AA (4.5:1 body, 3:1 large/UI). The pipeline SHALL assert that exactly one chromatic accent hue (--proven/--signal) plus the --pending warm tone are the only saturated colors in the palette, so that accent discipline is machine-enforced rather than a convention.
- **TOOL-03** `[P0]` The Rive asset pipeline SHALL validate every public/rive/*.riv against a committed manifest declaring its required state-machine name, inputs (e.g. requirement-proved triggers, gate-close state), and a content hash. WHEN a .riv is added or changed, the pipeline SHALL fail IF the file's state-machine/input names or hash do not match the manifest. WHERE prefers-reduced-motion is set OR WebGL/Rive fails to load, the loader SHALL render the committed static labeled-diagram fallback instead of the animation.
- **TOOL-04** `[P1]` The MDX pipeline SHALL validate every content/**/*.mdx file's frontmatter against a typed schema (title, slug, description, publishedAt, status, sources[]) and SHALL fail the build IF required fields are missing or malformed. WHEN MDX content contains a quantitative verification claim matching a configured pattern (e.g. "\d+/\d+ ... verified"), the claims-integrity linter SHALL require an adjacent evidence reference (source URL or evidence id) and SHALL fail the build IF the claim is unsourced.
- **TOOL-05** `[P0]` The repository SHALL enforce ESLint (incl. jsx-a11y and a custom no-raw-token rule), Prettier formatting, Stylelint, and `tsc --noEmit` in strict mode as deterministic gates. WHEN a developer attempts to commit, a pre-commit hook SHALL run format+lint on staged files and block the commit IF they fail. The same checks SHALL run in CI on the full tree, and CI SHALL be the authoritative gate (local hooks inform but the CI result decides mergeability).
- **TOOL-06** `[P0]` The visual-regression suite SHALL capture deterministic screenshots of the closed-loop at its named states (intent, proving, proved, gate-closed) and of each key route at mobile/tablet/desktop breakpoints, comparing against committed baselines with a bounded pixel-diff threshold. WHEN a captured frame differs beyond threshold, the visual gate SHALL fail and publish a diff artifact. WHILE prefers-reduced-motion is emulated, the suite SHALL assert the static fallback renders and is legible.
- **TOOL-07** `[P0]` The test suite SHALL run axe-core against every key route (including empty, loading, and error states of dynamic components) and SHALL fail the build IF any serious/critical violation is found (target: zero). WHEN a page contains the closed-loop or evidence-trace, the suite SHALL assert a text-equivalent/aria description exists. WHERE interactive controls exist, the suite SHALL assert full keyboard operability and visible focus states.
- **TOOL-08** `[P1]` The unit/component test layer SHALL test pure logic (token helpers, MDX utilities, evidence-trace formatting) and component rendering with Vitest + Testing Library. WHERE a component declares an animation, the test SHALL assert the animation is bound to a documented state input (a state→visual mapping) and SHALL fail IF an animation has no associated state. The suite SHALL cover loading, empty, and error states for any data-driven component.
- **TOOL-09** `[P0]` CI SHALL run tokens-drift, contrast, lint, typecheck, format, unit/coverage, Playwright visual, axe, Lighthouse/CWV budget, MDX/claims-integrity, link-check, and security/header checks as required status checks. IF any required check fails, THEN the merge gate SHALL block (fail-closed) and SHALL NOT allow deploy. The gate decision SHALL be computed only from these verifiable check outputs; no human or model self-report SHALL substitute for a passing check.
- **TOOL-10** `[P1]` DESIGN.md SHALL be the canonical specification of the token contract (names, values, semantics, accent discipline, typography, motion rules) and the token source files SHALL be derived from it. WHEN DESIGN.md or the token source changes, a governance check SHALL verify they remain consistent (same token names/values) and SHALL fail IF they diverge. The governance check SHALL be a required CI gate so the design contract cannot silently rot.
- **TOOL-11** `[P1]` CI SHALL run Lighthouse CI (or equivalent) against production-built key routes and SHALL fail IF performance, accessibility, best-practices, or SEO scores fall below committed thresholds (e.g. >=95) or IF LCP > 2.5s / CLS > 0.1 / INP budgets are exceeded. The pipeline SHALL enforce JS bundle and Rive asset size budgets and SHALL fail on regression beyond a documented delta.
- **TOOL-12** `[P0]` The deployment SHALL serve a strict Content-Security-Policy plus HSTS, X-Content-Type-Options, Referrer-Policy, frame-ancestors and Permissions-Policy headers, and CI SHALL verify these headers on a built/preview artifact. CI SHALL run dependency vulnerability scanning and secret scanning as required gates. IF a high/critical dependency vulnerability or a committed secret is detected, THEN the security gate SHALL fail and block deploy.
- **TOOL-13** `[P0]` The deployment pipeline SHALL build the Next.js App Router site as SSG/ISR and deploy it to the root domain autonomous-agent.dev only (no subdomain content). WHEN the main branch passes all required gates, the pipeline SHALL deploy an immutable, content-addressable build artifact and SHALL support one-click/atomic rollback to a prior build. IF the production deploy health-check fails, THEN the pipeline SHALL halt promotion and retain the previous live version.
- **TOOL-14** `[P1]` WHEN a pull request is opened or updated, the pipeline SHALL build and deploy an isolated preview environment at a unique URL and SHALL run the visual-regression, axe, Lighthouse, and header checks against that preview. The pipeline SHALL post the preview URL and a summary of gate results to the PR. WHERE a preview is created, it SHALL be access-controlled (noindex, optional auth) and SHALL be torn down when the PR is closed/merged.
- **TOOL-15** `[P1]` CI SHALL validate internal links, generated sitemap.xml, robots.txt, canonical tags, Open Graph/Twitter metadata, and JSON-LD structured data on built routes, and SHALL fail IF any internal link is broken or required SEO artifact/metadata is missing or invalid. The validator SHALL confirm no production route references a subdomain or sibling product.
- **TOOL-16** `[P1]` The font pipeline SHALL self-host Geist Sans and Geist Mono (woff2, subset, with font-display: swap and size-adjust to minimize CLS) and SHALL expose them via the token typography layer. The build SHALL fail IF a font file is missing, references an external host, or a machine-artifact component is not bound to the mono token. WHEN fonts load, the critical font SHALL be preloaded to protect LCP.

### PERF — Performance
- **PERF-01** `[P0]` WHEN any primary route ( / , /how-it-works, /proof, /manifesto ) finishes loading on a Moto-G-class mobile profile (4x CPU throttle, Slow-4G), the site SHALL meet field-grade Core Web Vitals thresholds: LCP < 2.5s, CLS < 0.1, and INP < 200ms at the 75th percentile.
- **PERF-02** `[P0]` WHERE the display (Geist Sans / Inter Variable), body, and mono (Geist Mono / JetBrains Mono) typefaces are used, the site SHALL self-host subsetted WOFF2 fonts, preload only the fonts needed for above-the-fold content, and apply font-display: optional (or swap with a metric-compatible fallback) so that no font swap shifts layout.
- **PERF-03** `[P0]` WHEN the homepage loads, the site SHALL defer the Rive runtime and the .riv asset until the centerpiece is near the viewport, render the animation on a worker/off the main thread where supported, and reserve its exact box so it never contributes to CLS.
- **PERF-04** `[P1]` WHERE raster or vector imagery is rendered, the site SHALL serve modern formats (AVIF/WebP with fallback), declare intrinsic width/height (or aspect-ratio), lazy-load below-the-fold images, and eagerly prioritize only the single LCP image if one exists.
- **PERF-05** `[P0]` The site SHALL enforce a per-route first-load JavaScript budget and code-split heavy, route-specific, and below-the-fold modules (Rive, Framer Motion, interactive proof trace, MDX-heavy content) behind dynamic imports so they are excluded from the initial bundle.
- **PERF-06** `[P1]` The site SHALL ship a per-route critical/initial CSS budget, generate styles from the Style-Dictionary design tokens (Tailwind v4) with unused utilities purged, and avoid render-blocking third-party stylesheets.
- **PERF-07** `[P1]` The site SHALL statically generate (SSG) all content routes at build time and use ISR only where content updates are expected (e.g. /writing), serving HTML from cache/CDN with no per-request server render on the critical path.
- **PERF-08** `[P0]` The site SHALL run Lighthouse-CI (or equivalent) plus bundle-size assertions on every pull request as a fail-closed gate, blocking merge IF performance score < 95, IF any CWV threshold is exceeded, or IF any JS/CSS/font/image/Rive budget is breached.
- **PERF-09** `[P0]` WHILE the visitor's prefers-reduced-motion is set to reduce, the site SHALL not download or initialize the Rive runtime/.riv, SHALL render the static labeled-diagram loop fallback instead, and SHALL disable Framer Motion non-essential animation — yielding equal-or-better CWV than the animated path.
- **PERF-10** `[P1]` WHERE any third-party or non-critical first-party script is included (privacy-respecting analytics, etc.), the site SHALL load it deferred/async or post-hydration, off the critical path, and SHALL keep the total third-party transfer and main-thread cost within a defined budget.
- **PERF-11** `[P1]` WHILE the visitor scrolls through any animated section, the site SHALL drive motion using compositor-friendly properties (transform/opacity), throttle scroll work via rAF/IntersectionObserver, and keep interaction latency within the INP budget.
- **PERF-12** `[P1]` WHEN the document is served, the site SHALL set correct resource-loading priorities — preconnect/dns-prefetch for any required origin, preload of the LCP resource and above-the-fold fonts, and explicit deprioritization (lazy/dynamic) of the Rive runtime, below-the-fold images, and non-critical scripts.
- **PERF-13** `[P1]` IF the site publishes any quantitative performance claim about itself (e.g. a Lighthouse score, an LCP figure, a bundle size) in visible content, THEN that claim SHALL be rendered as a monospaced machine artifact derived from a stored CI gate evidence record (metric, value, threshold, collected_at), and SHALL be suppressed or shown as pending IF no current verified evidence exists.
- **PERF-14** `[P1]` IF a non-critical asset (Rive .riv, an image, a deferred chunk) fails to load or is slow, THEN the site SHALL present a stable, reserved-space fallback (static diagram/poster, skeleton, or graceful omission) without layout shift, console-error storms, or a blocked main thread, and SHALL keep the page fully usable.
- **PERF-15** `[P2]` WHERE field telemetry is enabled, the site SHALL collect real-user Core Web Vitals (LCP, CLS, INP, TTFB) via a lightweight, privacy-respecting RUM path and attribute regressions to a route, without that telemetry itself harming performance.
- **PERF-16** `[P1]` WHERE the site is viewed at any breakpoint, the site SHALL serve responsively-sized assets, avoid loading desktop-only heavy media on mobile, and ensure breakpoint transitions and container queries produce no cumulative layout shift.

### A11Y — Accessibility & Reduced Motion
- **A11Y-01** `[P0]` The site SHALL render exactly one banner, one main, and one contentinfo landmark per page, with all interactive primary navigation inside a nav landmark; and WHERE a page contains sectioned content, the page SHALL maintain a heading hierarchy with no skipped levels (h1 -> h2 -> h3) and exactly one h1 per page.
- **A11Y-02** `[P0]` WHEN a visitor navigates with the Tab key, the site SHALL move focus through interactive elements in DOM/reading order with a focus indicator meeting WCAG 2.2 AA visibility, and WHEN focus first enters the page, the site SHALL expose a 'Skip to main content' link as the first focusable element that moves focus to the main landmark when activated.
- **A11Y-03** `[P0]` The site SHALL meet WCAG 2.2 AA contrast for all text and UI components: normal text >= 4.5:1, large text and graphical/UI-component boundaries >= 3:1; and IF a proposed token pairing falls below its threshold, THEN the design-token layer SHALL fail the contrast gate and that pairing SHALL NOT ship.
- **A11Y-04** `[P0]` WHILE the user agent reports `prefers-reduced-motion: reduce`, the closed-loop centerpiece SHALL render a static, fully labeled diagram of the loop (intent -> decompose -> implement -> verify -> prove -> fail-closed gate) showing the gate in its closed/holding state, and SHALL NOT autoplay the Rive state machine or any looping motion.
- **A11Y-05** `[P0]` WHILE `prefers-reduced-motion: reduce` is active, the evidence-trace panel, coverage-fill indicators, trace-reveal sequences, scrollytelling transitions, and all Framer Motion micro-interactions SHALL render in their final/resolved state immediately, with no fade, slide, stagger, pulse, or auto-advance.
- **A11Y-06** `[P0]` The site SHALL NOT use color as the only visual means of conveying a verification state, and WHERE a machine artifact or loop node communicates proven, pending, unproven/dim, or gate-closed status, the site SHALL pair the color with a redundant non-color cue (icon/glyph shape, text label, fill pattern, or state word in monospace).
- **A11Y-07** `[P0]` The site SHALL provide an accessible text equivalent for all animated and visual proof content: the closed-loop centerpiece SHALL expose an accessible name/description (or an associated visually-available text alternative) summarizing the six stages and the fail-closed outcome, and the evidence-trace records SHALL be exposed to assistive tech as readable text, not as an inert canvas.
- **A11Y-08** `[P0]` WHEN the accessibility CI job runs against every route in both the default and reduced-motion render paths, the build SHALL fail closed (non-zero exit, merge blocked) IF any axe-core violation of impact 'critical' or 'serious' is present, and the gate SHALL itself be evidence-backed (it records which rules ran and the result, never a self-asserted pass).
- **A11Y-09** `[P1]` WHEN content is zoomed to 200% (and up to 400% per WCAG 1.4.10) or the OS/browser text size is enlarged, the site SHALL reflow to a single readable column without two-dimensional scrolling and without clipping or overlapping text, including the monospace machine artifacts.
- **A11Y-10** `[P1]` The site SHALL give every form control and custom interactive control a programmatic accessible name and role, and WHEN the Get-notified submission is invalid or fails, THEN the site SHALL surface an accessible, text-based error associated with the field and announced to assistive tech, never conveyed by color alone.
- **A11Y-11** `[P1]` WHERE the site presents motion that starts automatically, runs longer than 5 seconds, or auto-updates (the closed-loop animation, any auto-advancing scrollytelling, the trace stream), the site SHALL provide a mechanism to pause, stop, or hide it, and the motion SHALL respect WCAG 2.2 SC 2.2.2 even when `prefers-reduced-motion` is not set.
- **A11Y-12** `[P1]` WHILE the closed-loop, self-hosted fonts, or trace content are loading, the site SHALL present an accessible, non-empty placeholder (the static labeled diagram and system fonts), and IF any of these assets fail to load, THEN the site SHALL keep the page operable and announce nothing misleading (no 'proven' state shown without its evidence text).
- **A11Y-13** `[P1]` The site SHALL NOT present any verification, coverage, or 'accessible/compliant' status as a published claim unless that claim is backed by a verifiable artifact, and IF an accessibility or coverage figure cannot currently be substantiated, THEN the site SHALL omit the specific number rather than assert it.
- **A11Y-14** `[P1]` The site SHALL preserve accessibility across responsive breakpoints: interactive targets SHALL meet a minimum size of 24x24 CSS px (WCAG 2.2 SC 2.5.8, target 44x44 where practical), content SHALL support both portrait and landscape (SC 1.3.4), and the loop SHALL degrade to the labeled diagram on small screens.
- **A11Y-15** `[P2]` WHEN a client-side route change completes (Next.js App Router navigation) or an in-page anchor is activated, the site SHALL move focus to a logical target (the main landmark or the relevant heading), update the document title to reflect the new page, and announce the page change to assistive technology.

### PRIV — Privacy, Analytics, Security & Consent
- **PRIV-01** `[P0]` The analytics layer SHALL operate as cookieless, privacy-first measurement that collects no personally identifying information, sets no cross-site or persistent first-party tracking identifier, and performs no fingerprinting. WHERE a visitor's data is recorded, the analytics layer SHALL store only aggregate, anonymized event counts (page, referrer, country, device-class) with no per-visitor identity.
- **PRIV-02** `[P0]` WHERE the analytics layer is genuinely cookieless and PII-free, the consent system SHALL NOT present a blocking cookie-consent modal or interstitial that gates content. WHEN any future feature would set a non-essential cookie or load a non-essential third-party script, THEN the consent system SHALL require prior opt-in consent before that cookie is set or that script loads.
- **PRIV-03** `[P0]` The consent system SHALL provide an accessible, persistent control to disable all optional analytics. WHEN a visitor opts out, THEN the analytics layer SHALL stop emitting all measurement events for the remainder of the session and on subsequent visits, and the System SHALL record the opt-out as a first-party, strictly-necessary preference only.
- **PRIV-04** `[P1]` WHERE the optional 'Get notified' email capture is presented, the email-capture component SHALL collect only an email address, SHALL require an explicit affirmative action with an adjacent purpose statement, and SHALL NOT pre-tick consent or bundle it with unrelated processing. WHEN an email is submitted, THEN the System SHALL transmit it over HTTPS to a single declared processor and SHALL confirm submission without exposing the address to client-side third parties.
- **PRIV-05** `[P0]` The edge/server SHALL serve every response over HTTPS and SHALL set the security headers: Strict-Transport-Security (with a long max-age and includeSubDomains), X-Content-Type-Options: nosniff, Referrer-Policy: strict-origin-when-cross-origin, X-Frame-Options: DENY (or CSP frame-ancestors 'none'), Permissions-Policy denying camera/microphone/geolocation by default, and Cross-Origin-Opener-Policy: same-origin. WHEN an HTTP request arrives, THEN the edge SHALL 308-redirect it to HTTPS.
- **PRIV-06** `[P0]` The System SHALL enforce a Content-Security-Policy that defaults to `default-src 'self'`, allows only the explicitly enumerated sources required by the Rive runtime, self-hosted fonts, and the privacy-first analytics endpoint, sets `object-src 'none'`, `base-uri 'self'`, and `frame-ancestors 'none'`. WHERE inline scripts or styles are unavoidable, the CSP SHALL authorize them via per-response nonces rather than `unsafe-inline`.
- **PRIV-07** `[P1]` WHERE the site loads any script or style from a non-same-origin source, the System SHALL apply Subresource Integrity (SRI) hashes and `crossorigin` attributes. The System SHALL prefer self-hosting for fonts, the Rive runtime, and analytics so that the third-party surface is as close to zero as practical.
- **PRIV-08** `[P1]` WHEN the browser presents a Global Privacy Control (`Sec-GPC: 1`) or `DNT: 1` signal, THEN the analytics layer SHALL suppress all optional measurement for that visitor by default without requiring an explicit in-page opt-out, and the consent system SHALL reflect the honored state in any privacy UI.
- **PRIV-09** `[P0]` The System SHALL publish a Privacy page that accurately describes all data processing (cookieless analytics, optional email capture, any sub-processors), the lawful basis under GDPR, the categories under CCPA/CPRA, retention, and the contact route for data-subject requests. The System SHALL keep the published policy in exact correspondence with the site's actual data behavior.
- **PRIV-10** `[P1]` The footer SHALL present a light trust/security note, rendered in the monospace machine-artifact style, that states only verifiable, presently-true facts about the site's posture (cookieless analytics, hardened transport, no sale of personal data). The footer note SHALL NOT publish any unverifiable count or claim, in keeping with the claims-integrity guardrail.
- **PRIV-11** `[P1]` IF the analytics endpoint is blocked, fails, or times out, THEN the System SHALL fail silently with no console error surfaced to the visitor, no retry storm, and no impact on page interactivity or Core Web Vitals. IF the email-capture submission fails, THEN the email-capture component SHALL show an inline, accessible error state and SHALL NOT lose the entered value.
- **PRIV-12** `[P1]` The consent and privacy UI SHALL meet WCAG 2.2 AA: keyboard operable, visible focus, AA contrast, and respectful of `prefers-reduced-motion`. WHERE any privacy UI animates (e.g. a notice slide-in or toggle transition), the motion SHALL be informational and SHALL have a static, legible reduced-motion fallback.
- **PRIV-13** `[P1]` The email-capture API SHALL validate and normalize input server-side, rate-limit by IP/identifier, and reject malformed or disposable-abuse submissions, using a privacy-respecting anti-abuse mechanism (e.g. honeypot + a privacy-preserving challenge such as a proof-of-work or Cloudflare Turnstile) rather than a tracking-based CAPTCHA. WHEN abusive volume is detected, THEN the API SHALL throttle without exposing whether a given address already exists.
- **PRIV-14** `[P0]` The System SHALL include an automated privacy/security verification suite that asserts the live behavior matches every published privacy/security claim (cookieless, headers present, CSP enforced, GPC honored, no PII leaked to analytics). WHERE any published claim cannot be verified by the suite, the System SHALL block that claim from publication until it can be evidenced.

# PART C — Coverage & Adversarial Audit Report

**Five independent adversarial lenses.** Severity totals: **P0=11 · P1=28 · P2=17 · P3=8.** Within-domain traceability is 100%; the central finding is that domains were authored in isolation, so cross-domain integration carries the defects (resolved in Part D).

## Lens: coverage-traceability (coverage=100)

Internal requirement→design→task traceability is fully clean: all 189 requirements across the 12 domains (DS, IA, HOME, LOOP, CONTENT, PAGE, TECH, TOOL, PERF, A11Y, SEO, PRIV) carry at least one design decision AND at least one task, with zero orphan design items and zero orphan tasks — so within-domain coverage is 100%. The decisive defect is in the cross-domain dependency graph: the entire HOME domain (all 15 requirements) was authored against five neighbor-domain ID namespaces — TOKENS-*, MOTION-*, PROOF-*, CLAIMS-*, NAV-* — that do not exist anywhere in the delivered spec, producing 23 dangling dependsOn edges; only the HOME references to LOOP-01/02, A11Y-01, CONTENT-01, and PERF-01 resolve, so just ~30% of HOME's cross-domain edges point at real requirements. This means the HOME domain cannot be ordered or built from its stated dependencies without first remapping every alias to the canonical owners (DS for tokens/motion, IA for nav, CONTENT-06 for claims, LOOP-05/PAGE-04 for proof). Compounding this, several genuinely shared concerns (/writing route ownership, the Get-notified affordance, claims integrity) are independently re-specified in 4–9 domains with no dependsOn links binding the UI surfaces to their authoritative backend/owner requirements, and the spec has no machine-checkable referential-integrity gate to catch any of it. Verdict: per the 100%-coverage bar, the spec PASSES on orphan-requirement and orphan-work coverage but FAILS cross-domain dependency integrity — the HOME dependency block must be rewritten against real IDs and a dependsOn/reqId resolver added to CI before the traceability matrix can be considered sound.

- **[P0][sync]** _HOME domain — all 15 requirements (HOME-01 through HOME-15) dependsOn arrays_ — The entire Homepage Scrollytelling domain was authored against five cross-domain ID namespaces that do not exist anywhere in the delivered spec: TOKENS-* (TOKENS-01), MOTION-* (MOTION-01, MOTION-02), PROOF-* (PROOF-01), CLAIMS-* (CLAIMS-01), and NAV-* (NAV-01). These yield 23 dangling dependency edges (TOKENS-01 x5, MOTION-01 x7, MOTION-02 x4, PROOF-01 x1, CLAIMS-01 x4, NAV-01 x2) touching every single HOME requirement. The actual delivered domains use entirely different prefixes — the design system is DS-*, navigation is IA-*, the proof page is PAGE-*/the loop is LOOP-*, claims integrity is CONTENT-06, motion lives in DS-08/DS-09. So the HOME spec's dependency graph is a reference to a sibling-spec taxonomy that was never reconciled with the one actually shipped. A traceability matrix built from these IDs cannot resolve a single HOME upstream cross-domain dependency to a real requirement.  
  **Fix:** Remap every HOME cross-domain dependsOn to a real existing ID before this spec is treated as buildable: TOKENS-01 -> DS-01 (token source of truth); MOTION-01 -> DS-08 (motion-as-information tokens) and MOTION-02 -> DS-09 (reduced-motion) [or split per intent]; PROOF-01 -> PAGE-04 (interactive evidence trace) or LOOP-05 (EvidenceTrace panel) — pick the canonical owner; CLAIMS-01 -> CONTENT-06 (claims-integrity guardrail); NAV-01 -> IA-03 or IA-04 (no-funnel nav / responsive nav). ## Lens: tier1-standards-principles (coverage=93)

All five mandated Tier-1 guardrails ARE enforced somewhere as concrete, testable requirements — this spec is unusually disciplined and best-in-class on the gate-rejection and one-accent rules. But the audit found real defects: (1) a cross-domain CONTRADICTION on the corrected formal-verification number — the corpus now records a self-counted 32/32 (the 21/21 being the deprecated figure), yet TOOL/TECH denylist "21/21" while CONTENT-08's open question and the brief itself (which predates the correction) instruct publishing NO count; the denylist targets the wrong string and no domain pins the canonical decision, so the guardrail can simultaneously pass CI and ship an out-of-policy claim; (2) the motion-as-information rule is enforced per-domain but has NO single owner and its "no decorative loop" check is keyword-grep, defeated by a Rive-internal or inline-style loop; (3) one-accent density is a P0 rule with only a "heuristic/documented fraction" — no concrete ceiling, so it is not machine-enforceable; (4) the monospace inverse ("no marketing prose in mono") is audit-only, weaker than the forward rule; (5) the satisfying-gate "no red" audit is frame-sampled and can miss a sub-threshold red between sampled frames, and the accent-area bound is undefined. Coverage of the five named rules is ~93%: each is present and mostly fail-closed, but the claims-integrity contradiction is a P0 that undermines the brand's own north-star, and three rules fall short of "machine-enforced, not convention" by a concrete-threshold gap. Quality verdict: STRONG with one P0 contradiction and several P1 enforcement-tightening gaps to reach genuinely best-in-class.

- **[P0][contradiction]** _TOOL-04 / T-TOOL-12 / TECH-13 / SEO-13 / CONTENT-06 / DS-16 vs PRD §0.1.5 and brief §7_ — The claims-integrity guardrail contradicts itself across domains on WHICH number is forbidden. The product corpus (PRD_MERGED §0.1.5, §3) corrected the count from a DEPRECATED 21/21 to a reproducible, self-counted 32/32; 21/21 is the wrong/legacy figure and 32/32 is the corrected canonical one. Yet TOOL-04 acceptance and T-TOOL-12 hard-deny the string '21/21', and the TOOL design narrative ('21/21 corrected to 32/32') treats 21/21 as the thing to block — meaning a literal '32/32 formally verified' (the CORRECTED but still site-unverifiable number) would pass the TOOL denylist. Meanwhile the website brief (§7), which predates the correction, says publish NO specific 'N/N formally verified' number at all, and CONTENT-08's open question explicitly flags this unresolved. No domain pins the canonical site decision. Result: a build could pass the claims gate while shipping '32/32 formally verified' — exactly the out-of-policy, site-unverifiable claim the guardrail exists to stop. For a proof company whose north-star is 'no self-report accepted as proof,' shipping a self-counted count the SITE cannot independently verify is a brand-fatal defect.  
  **Fix:** Add one canonical, cross-domain claims-policy constant (e.g. content/claims-registry.json with a binding note in DESIGN.md) that registers the formal-verification claim as verifiable:false and forbids ANY 'N/N (formally )?verified' / 'N/N checked' pattern regardless of the specific N — not just the literal 21/21. The denylist regex must be value-agnostic (\b\d+\s*/\s*\d+\s+(formally\s+)?(verified|checked)\b) so 32/32, 21/21, and 100/100 all fail equally. Resolve CONTENT-08's open question to the brief's posture: ship a qualitative 'machine-checked formal verification of the requirement logic model (Z3)' with no number until a site-specific, independently-reproducible re-verification exists. Make CONTENT-06/TECH-13/SEO-13/TOOL-04/PRIV-14/LOOP-13/DS-16/A11Y-13 all consume the same registry so the rule cannot diverge.
- **[P1][standards]** _DS-08 / HOME-14 / TECH-06 / LOOP-02 / PAGE-12 / TOOL-08 — motion-as-information ownership_ — The motion-as-information rule (the HARD art-direction rule: every animation is a state machine, not a decorative loop) IS enforced, but redundantly and inconsistently across six domains, each with its own registry concept (DS-08 'motion-intent registry', TECH-06 'lib/motion/registry.ts', LOOP-02 'state-machine contract', PAGE-09 'motion contract', TOOL-08 'components/motion/registry.ts'). Nothing designates a single canonical registry that all animated components must appear in. A Framer component registered in TECH's registry but absent from PAGE's, or a Rive-internal animation that never surfaces to any registry, can ship without any single check catching it. This is the spec's most load-bearing brand rule and it has no single owner — a best-in-class gap.  
  **Fix:** Designate ONE canonical motion-intent registry (own it in the Brand & Design System domain, DS-08) as the single source of truth that TECH-06, LOOP-02, PAGE-12, HOME-14, and TOOL-08 all import and assert against. The CI motion-integrity test should fail if ANY animated element (Framer animate/variants, CSS keyframe, OR Rive state-machine state) is absent from that one registry, so the rule is enforced once, globally, not five times locally.
- **[P1][untestable]** _DS-08 AC3 / HOME-14 AC3 / TECH-06 / PAGE-12 — 'no decorative/infinite loop' detection method_ — The mechanism for detecting a banned decorative loop is a code grep for 'animation: ... infinite' / 'repeat: Infinity' / unbounded keyframes (DS-08 AC3, TECH-06 design, HOME-14 AC3). The brief's hard rule ('no AI-glow') targets exactly the kind of ambient motion most likely to live inside Rive, where the grep is blind.

# PART D — Audit resolutions (every P0 + the key P1s closed)
