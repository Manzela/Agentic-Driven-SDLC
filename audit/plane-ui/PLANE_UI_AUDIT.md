# Plane UI — Forensic Audit & Linear/Notion Light-Theme Enhancement Spec

**Date:** 2026-06-16 · **Target:** live self-hosted Plane v1.3.1, project `ASCP` (`http://localhost:8090`).
**Method:** Playwright capture of the live board + tickets → 16-agent fan-out (Plane-docs crawl + Linear/Notion/light-theme/a11y research + 6-dimension forensic audit) → synthesis.
**Captures:** `audit/plane-ui/screens/` (01 list · 02 kanban · 03 ticket · 04 themed · 05 colored-states) · tokens `audit/plane-ui/design-tokens.json`.

## Executive summary

The live Plane board (ground truth: audit/plane-ui/design-tokens.json + screens/01-issues-default.png, 02-kanban-board.png, 03-ticket-detail.png) is stock Plane in its default light theme, and it actively works against this project's whole value proposition: a Spec-to-Evidence control plane whose entire reason to exist is that humans AND machines can see, at a glance, where every coverage item sits across a 12-state agent lifecycle (Backlog → Agent-Triaged → Spec-Compiling → Spec-Verified → Plan-Approved → Agent-Executing → In-Verification → Human-Review → Done, plus terminal HANDOFF/Blocked/Failed) and whether it is proven, on what Evidence_Record, by which agent, against which requirement. The board renders almost none of that. The four diagnostic root causes: (1) COLLAPSED SURFACE + DEAD TOKENS — body bg oklch(0.9543 0.001 230.67) is a flat cool gray ~1% off the pure-white cards, radius:0px, shadow:none, and the `border` token is mis-set to the TEXT color, so cards have zero figure-ground and read as a spreadsheet, the opposite of Linear-crisp/Notion-calm. (2) SEMANTICS CARRIED BY COLOR ALONE AND FAILING WCAG — workflow state is a neutral gray dot (#4e5355, zero hue: status is literally not color-coded), and the only colored chrome is the saturated `--editor-colors-*` pastel palette whose label-text-on-pastel pairs compute 1.8–2.3:1 (orange #fd9038 = 2.28:1, green #0fc27b = 2.33:1, light-blue #17bee9 = 2.19:1 on white) — failing 1.4.3, 1.4.11, and 1.4.1 simultaneously; status is unreadable to color-blind humans and unparseable to agents. (3) THE MACHINE PAYLOAD IS BURIED OR INVISIBLE — evidence_status appears NOWHERE on any card/row/detail despite being the single most important field; agent_role is swallowed into a "4 Labels" overflow pill; requirement_id (ASCP-E3-5) lives only inside the H1 title string next to the unrelated sequence ID (ASCP-24); the four-field Evidence_Record and EARS modality tags ([ubiquitous]/[event-driven]/[unwanted]) render as undifferentiated bold prose. (4) DENSITY INVERTED — Kanban cards waste a full row on a status pill that merely restates their own column, another row on empty ghost-icon placeholders, then truncate the title (the actual content) to one line; only ~3.5 of 12 columns fit at 1512px with no overview, so the pipeline is never legible end-to-end. The fix is overwhelmingly a token + view-config + content-convention effort, NOT a fork: a luminance-first OKLCH light theme (3-surface warm-neutral ramp, contrast-verified text ladder with a hard 4.5:1 floor, one restrained indigo accent reserved for interaction only, and a 12-hue equal-OKLCH-lightness status palette that doubles as label-text AND status-glyph color), delivered by overriding Plane's CSS custom properties; paired with Plane self-hosted Community config that already supports most of what's needed (custom States in pipeline order, the 10 custom fields + 22 labels the blueprint already defines, per-project public Saved Views with minimal display properties, board grouping/sub-grouping). The decisive reconciliation finding: the blueprint ALREADY models agent_role/evidence_status/requirement_id/coverage_type/output_hash/ears_pattern/gate as 10 first-class custom fields — so the real defect is not the data model, it is that the board UI never SURFACES those typed fields on the card face; the highest-leverage move is therefore a view-config + CSS pass that promotes the existing custom fields (especially evidence_status) to the most prominent, color-encoded, machine-readable token on every card. Two hard Community-edition boundaries to design around: Plane has NO native WIP limits and NO native transition workflows (both must be enforced via webhooks/REST, which the project already does in webhook_handler.py / plane_client.py), and there is no in-product themer beyond Light/Dark — the crisp restyle is a self-hosted CSS-variable override layer, kept separate so it survives upgrades.

## Top findings

- EVIDENCE_STATUS IS INVISIBLE EVERYWHERE — the #1 machine/human field is absent from every card, row, and the detail Properties panel. The EARS body of ASCP-24 mandates a {unproven→proven→failed} status machine and a 4-field Evidence_Record, and the blueprint ALREADY defines `evidence_status` as a custom dropdown field, yet the UI shows only Plane's generic State='Done'. A reviewer cannot distinguish 'Done-but-unproven' from 'Done-and-proven', and the autonomous loop has no rendered field to read. Highest-leverage single fix: surface evidence_status as the most prominent color-coded chip on every card (proven=green, unproven=amber, failed=red), reusing the existing field — no new data model needed.
- STATUS IS NOT COLOR-CODED AND FAILS WCAG ON EVERY AXIS — the live state 'dot' is neutral gray #4e5355 (zero saturation), so all 12 lifecycle states look identical; the only colored chrome (the `--editor-colors-*` pastels) produces label-text-on-tint contrast of 1.8–2.3:1 (orange 2.28:1, green 2.33:1, light-blue 2.19:1 on white), failing 1.4.3 (text), 1.4.11 (non-text), and 1.4.1 (color-alone) at once. Status must become a shape+icon+text+color quad: a distinct status glyph per state, a text label, and one equal-OKLCH-lightness hue per state that passes 4.5:1 as text AND 3:1 as a glyph.
- COLLAPSED SURFACE RAMP + MIS-TOKENED BORDER — body bg oklch(0.9543 0.001 230.67) sits ~1% off pure-white cards (cards on a ~1.0:1 board = no figure-ground), radius:0px reads as a database tool not a refined PM surface, shadow:none removes Linear/Notion elevation, and the `body.border` token is wrongly set to the near-black TEXT color oklch(0.2378). Replace with a 3-surface warm-neutral ramp (canvas #FBFBFA / surface #FFFFFF / sunken #F6F6F4), a real hairline border token (~#E8E8E4), 6–8px radius, and border-first elevation with one whisper shadow.
- THE MACHINE PAYLOAD IS SMUGGLED THROUGH FREE-TEXT AND OVERFLOW — agent_role renders only as a label that gets swallowed into a '4 Labels +N' overflow pill; requirement_id (ASCP-E3-5) lives only inside the H1 title string, visually unrelated to and confusable with the sequence ID (ASCP-24); the four-field Evidence_Record, EARS modality tags, and encoded identifiers (feature_list.json, output_hash, four-field gate) all render as undifferentiated bold prose with no mono/chip treatment. The board answers the wrong questions at a glance and forces both humans and scrapers to NLP-parse prose.
- REDUNDANT-PILL + GHOST-ICON DENSITY INVERSION — every Kanban card spends a full row on a state pill that merely restates the column it already sits in ('Backlog' inside Backlog), another row on empty monochrome ghost-icons (estimate/date/assignee placeholders with no values), then truncates the title — the actual EARS content — to one line at ~38 chars. Net: ~7 low-signal cards per viewport where ~12 high-signal cards should fit. Hide redundant state in board view, render meta only when populated, and allow a 2-line title clamp.
- 12-COLUMN PIPELINE IS NEVER LEGIBLE END-TO-END — at 1512px only ~3.5 of the 12 lifecycle columns render (~330px each) with no minimap/overview, so the downstream half of the agent lifecycle is invisible without scrolling; column headers carry a raw integer count only (Backlog 186 vs single-digit active stages) with no WIP signal, no color band, and no count at all on empty stages, so the board's headline fact — a 30x back-loaded, downstream-starved pipeline — has zero visual encoding. Narrow columns to ~272px, add a sticky lifecycle overview strip, and encode WIP/aging in headers.
- NO NON-COLOR ENCODING AND NO MACHINE-READABILITY CONTRACT — beyond Plane's default DOM there are no stable data-* hooks (data-state, data-evidence-status, data-agent-role, data-requirement-id, data-trace), no explicit ARIA list/listitem column/card roles, and no parallel structured projection. For an agent-driven board the highest-leverage transparency win is a documented contract: dual-encode every status (color+glyph+text), emit canonical data-* attributes + ARIA, and regenerate a /board.json from the REST API so agents read typed fields instead of scraping pixels.
- PLANE COMMUNITY HAS NO NATIVE WIP LIMITS AND NO NATIVE TRANSITION WORKFLOWS — the two governance primitives this control plane depends on (WIP caps per state, evidence-gated transitions, human-only Plan-Approved/Done edges) do not exist as Community config; native Workflows/Approvals are Business/Enterprise-gated. They must be enforced via webhooks + REST (which the project already does in webhook_handler.py and the actor-authority guards in plane_client.py), and visualized via header WIP badges + an evidence-status chip rather than expected as a settings toggle.

## Content conventions (max human + machine legibility)

- EVERY card/row/detail surfaces evidence_status as the leading, color-coded chip: proven=green check + test count (e.g. '✔ 4/4'), unproven=amber half-glyph, failed=red. This is the single most important glanceable signal on a Spec-to-Evidence board and the field already exists in the data model — the convention is that it is always rendered, always first, never overflow-hidden.
- Pin agent_role as the second meta token with its own short uppercase mono code chip, excluded from the generic 'N Labels' overflow, so the owning agent per ticket is readable at board density even though the column already implies a pipeline stage.
- Split structured identity from prose in titles: render requirement_id (e.g. ASCP-E3-5) and epic ref as a small mono eyebrow above the title, distinct from Plane's sequence ID (ASCP-24); keep the H1 to the human-readable name only; move the four design constraints out of the title into the EARS body (single source of truth, no drift).
- Render EARS modality keywords as fixed-width color-coded mono chips in a left gutter, not bold prose: ubiquitous=gray, event-driven=blue, state-driven=green, unwanted=red/peach, optional=purple — so requirement modality is a scannable left rail for humans and a trivial selector for machines, and the existing color palette finally encodes meaning.
- Wrap all encoded/machine-load-bearing identifiers in inline mono code style: feature_list.json, Evidence_Record, output_hash, test_file, test_name, collected_at, nfr_subtype, and enum/default literals ({functional,NFR,WIRING}, DEFAULT true). Disambiguates underscores from word breaks and gives parsers and humans one consistent visual class for 'this is a literal field/file/flag'.
- Number and anchor every EARS requirement (R1…Rn) and acceptance criterion (AC1…ACn) with stable deep-link ids (#r3, #ac2), and add an explicit 'Verifies: R2,R4' / 'Verified by: AC1,AC3' traceability spine plus a coverage strip ('N requirements · M criteria · K uncovered') so coverage is auditable at a glance and citable in commits/comments/agent logs.
- Render acceptance criteria as a live verification ledger: each AC gets a status pill (pending/passing/failing/blocked) and a 4-cell Evidence_Record strip (test_file/test_name/output_hash/collected_at) that turns green only when all four fields are non-empty — making the four-field evidence gate's state observable directly on the criterion rather than asserted in prose.
- Adopt a fixed, parseable Evidence_Record comment template the verifier agent always posts and pins: a fenced ```evidence block with exactly the four mandated keys in fixed order (test_file, test_name, output_hash, collected_at) plus evidence_kind, so the PreToolUse gate and human readers can deterministically validate 'four non-empty fields'.
- Make coverage_type single-valued per ticket (functional XOR NFR XOR WIRING) via the typed custom field, eliminating the current contradictory type:wiring + type:functional + type:nfr label trio that gives any parser three conflicting answers.
- Emit stable data-* attributes + ARIA on every rendered element as the machine-readability fallback to the typed REST fields: data-state (canonical enum, not display text), data-state-group, data-evidence-status, data-evidence-tests='4/4', data-agent-role, data-requirement-id, data-coverage-type, data-priority, data-blocked, data-trace-spec/test/output; column=role=list aria-label='<state> (<count>)', card=role=listitem with a summarizing aria-label, terminal states aria-disabled, status glyph aria-label not just visual. Document this attribute contract so agents/tests/observability target fields deterministically.
- Maintain a parallel /board.json (optionally JSON-LD) regenerated from the Plane REST API carrying the same canonical enums, Evidence_Record, trace ids, and transition history — agents consume the JSON, the DOM data-* attributes are the visual-debug oracle, humans get the styled board: one source of truth, three faithful projections.
- Group the detail Properties rail by namespace (Type, Phase, Gate, Agent, Priority) with hairline separators instead of an unsorted colored chip pile, and collapse empty native fields (Assignees/Start date/Due date) into a single muted 'Add field' affordance so populated facts dominate.
- Order board columns strictly by SDLC flow (Backlog → Agent-Triaged → Spec-Compiling → Spec-Verified → Plan-Approved → Agent-Executing → In-Verification → Human-Review → Done; terminal HANDOFF/Blocked/Failed grouped at the right) and give empty downstream stages a real empty-state ('No items in <stage>' + what enters it + count badge 0) so an empty queue reads as meaningful signal, not a broken/misconfigured column.

## Per-dimension findings (16 reports)


### Plane Kanban/board layout: grouping, sub-grouping, swimlanes, column config, the 5 layouts, WIP, ordering, and card display properties (self-hosted Community edition)

Self-hosted Community Plane (the free, no-license-key, AGPLv3 edition) has feature parity with Cloud's Free tier, which means the entire board/layout toolkit relevant to this task is available: all 5 layouts (List, Board/Kanban, Calendar, Table/spreadsheet, Timeline/Gantt), Group by, Sub-group by (the practical equivalent of swimlanes), Order by, Show empty groups, Show sub-work items, and the full set of card Display properties. Board columns are NOT independently configured — they are a pure projection of whatever you Group by. Grouping by State yields workflow columns (Backlog/Todo/In Progress/Done/Cancelled), and that column ORDER is governed by the state order in Project Settings > States, so the board is configured indirectly through states, display options, and views rather than a dedicated board-column editor. Two real constraints exist for the goal: (1) Plane has NO native WIP limits anywhere — there is no per-column card cap, so WIP must be enforced by convention/automation, not configuration; (2) private (per-user) views and estimate systems beyond 6 values require Pro/Com

**Recommendations:**
- **Design the board through a tight State taxonomy, not a column editor** (workflow-config) — Because board columns are a projection of Group by State, the column set IS the state set. For a crisp Linear-style SDLC pipeline, define states at Project > Settings > States in pipeline order, one per group, e.g. Backlog (Backlog) -> Tria  
  _https://docs.plane.so/core-concepts/issues/states (Project > Settings > States)_
- **Use Sub-group by sparingly as deliberate swimlanes** (workflow-config) — Plane's swimlanes = Sub-group by. The highest-readability default is Group by State with NO sub-group (single clean pipeline). Offer a second saved View 'By Owner' = Group by State + Sub-group by Assignee for standup/load-balancing, and opt
